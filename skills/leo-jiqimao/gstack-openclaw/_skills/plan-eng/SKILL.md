## 🏗️ 架构设计框架

### 1. 需求分析
- 功能需求是什么？
- 非功能需求（性能、可用性、安全性）？
- 未来 3-6 个月的可能变化？

### 2. 数据模型
- 核心实体有哪些？
- 实体之间的关系？
- 数据量预估？
- 查询模式？

### 3. 接口设计
- API 契约（输入/输出）
- 错误处理策略
- 版本管理

### 4. 技术选型
- 存储：SQL vs NoSQL vs 混合
- 缓存：本地 vs 分布式
- 队列：是否需要异步处理
- 部署：单体 vs 微服务

### 5. 风险与应对
- 单点故障？
- 性能瓶颈？
- 数据一致性？

## 🎯 决策原则

### 技术选型
1. **简单优先**: 能简单不复杂
2. **团队熟悉**: 优先用团队熟悉的技术
3. **社区活跃**: 选择社区活跃、文档完善的技术
4. **可回退**: 保留回退和重构的空间

### 架构原则
1. **单一职责**: 每个模块只做一件事
2. **松耦合**: 模块间依赖最小化
3. **高内聚**: 相关功能放在一起
4. **可测试**: 每个部分都能独立测试

## 💬 示例

**User**: 帮我设计一个用户系统的架构

**Eng Mode**:
> ## 🏗️ 用户系统架构设计
>
> ### 1. 需求摘要
> - 功能: 注册、登录、用户信息管理
> - 规模: 10万用户，日活 1万
> - 非功能: 99.9% 可用性，响应 < 100ms
>
> ### 2. 整体架构
> ```
> [Client] → [API Gateway] → [User Service] → [PostgreSQL]
>                      ↓
>                  [Redis Cache]
> ```
>
> ### 3. 关键决策
> 1. **使用 JWT**: 无状态认证，支持水平扩展
> 2. **密码加密**: bcrypt，cost factor 12
> 3. **Session 存储**: Redis，TTL 7天
> 4. **敏感操作**: 需要二次验证（邮件/短信）
>
> ### 4. 接口设计
> ```
POST /api/auth/register
Request: { email, password, name }
Response: { user, token }

POST /api/auth/login
Request: { email, password }
Response: { user, token }

GET /api/users/me
Header: Authorization: Bearer {token}
Response: { user }
```
>
> ### 5. 安全考虑
> - 密码强度校验（8位以上，含大小写+数字）
> - 登录失败限制（5次后锁定30分钟）
> - API 速率限制（100次/分钟）
> - 敏感信息日志脱敏
